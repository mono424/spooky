sp00ky CRDT — how it works (and where it's going)

Three pictures: the old bypass, the target architecture, and what's actually in the code today.

1. Old design: the bypass being removed

CRDT updates skipped the entire sync pipeline. They wrote directly to a sidecar table, ran their own LIVE channel, and never touched DBSP, version arrays, or row-level auth.

Why this was weird

IssueWhy
Two LIVE channels per recordOne on _00_list_ref (normal), one on _00_crdt (CRDT). No coordination.
Open permissions_00_crdt table was WHERE true. Any client could write any field.
Snapshot overwriteConcurrent edits within a 300ms debounce clobbered each other.
Echo by wall clock500ms timestamp heuristic, fragile under reconnects.
Cursors squattingEphemeral cursor data shared the persistent _00_crdt table.
Invisible to DBSPCRDT content lived outside the reactive query graph.

2. New design: column-on-record + native Loro on the SSP

_00_crdt becomes a JSON column on the record itself (the CLI was already injecting it — runtime just didn't use it). Updates flow through the normal UpQueue. The SSP holds in-memory LoroDocs and does real CRDT merges. Down-sync rides the existing _00_list_ref pipeline.

5. Things to keep in mind

  • Per-keystroke _00_rv bump. Every applied update triggers a record refetch on every observer. Validate under multi-client load; coalesce on the server if chatty.
  • Cross-SSP races. Multiple SSPs writing the same (record, field) can race the read-modify-write. Mitigations: consistent-hash routing in the scheduler, or a CAS on rev in the SurrealDB UPDATE.
  • Loro version drift. Server loro crate (Rust) and client loro-crdt (npm) must stay on compatible majors. Pinned to v1.
  • DBSP filter visibility. Storing snapshots in _00_crdt doesn't make queries like "threads where title contains X" work — title is opaque bytes there. A derived plain-text mirror could be added later.
