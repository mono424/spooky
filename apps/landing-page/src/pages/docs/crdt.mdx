export const hookCode = `import { useCrdtField, useDb, useQuery } from '@spooky-sync/client-solid';
import { CollaborativeEditor } from './CollaborativeEditor';

export function ThreadEditor(props: { threadId: string }) {
  const db = useDb();
  const threadResult = useQuery(() =>
    db.query('thread').where({ id: \`thread:\${props.threadId}\` }).one().build()
  );
  const thread = () => threadResult.data() ?? null;

  const titleField = useCrdtField(
    'thread',
    () => thread()?.id,
    'title',
    () => thread()?.title
  );

  const handleTitleChange = async (value: string) => {
    const t = thread();
    if (!t?.id) return;
    await db.update('thread', t.id, { title: value }, { debounced: true });
  };

  return (
    <CollaborativeEditor
      field={titleField()}
      content={thread()?.title}
      placeholder="Untitled"
      singleLine
      onUpdate={handleTitleChange}
    />
  );
}`;

CRDT Fields let multiple clients edit the same field concurrently without conflicts. Sp00ky uses [Loro](https://loro.dev) as the CRDT engine and syncs state in real time through a dedicated `_00_crdt` table — open a document on two devices and every keystroke merges cleanly, including cursor presence.

## Activating CRDT on a Field

Mark any string field as CRDT-enabled by adding a `-- @crdt <type>` comment directly above its `DEFINE FIELD` statement in your `.surql` schema:

Supported CRDT types:

| Type | Use Case |
|---|---|
| `text` | Rich or plain text with character-level merging (fully supported) |
| `map` | Key/value structures with last-write-wins per key |
| `list` | Ordered collections with stable insertion |
| `counter` | Monotonic numeric counters |

After adding or changing a `@crdt` annotation, run `spky` to regenerate your types and migrations so the CRDT metadata flows into your generated schema.

<Note>
  The annotation is a comment — SurrealDB ignores it, but the Sp00ky parser picks it up and attaches the CRDT type to the field's `ColumnSchema` entry.
</Note>

## Using CRDT Fields in SolidJS

Two pieces work together: the `useCrdtField` hook binds a `CrdtField` to a record/field pair, and `CollaborativeEditor` wires that field into a TipTap editor with Loro sync, undo, and remote cursors.

### useCrdtField Arguments

- **`table`** — The table name (must have a `@crdt` field).
- **`recordIdAccessor`** — A reactive getter returning the record ID (e.g. `thread:abc`), or `undefined` while the record isn't ready. The hook rebinds automatically when this changes.
- **`field`** — The CRDT-enabled field name on the table.
- **`fallbackAccessor`** — A reactive getter returning the current non-CRDT value. Used to seed the document for records that existed before the field was marked as CRDT.

### CollaborativeEditor Props

- **`field`** — The `CrdtField` from `useCrdtField()`.
- **`content`** — Fallback text for records without saved CRDT state yet.
- **`placeholder`** — Placeholder text when empty.
- **`singleLine`** — Disables block nodes and intercepts `Enter` for title-style inputs.
- **`username`** — Label attached to this peer's cursor in remote-presence rendering.
- **`onUpdate(text)`** — Fired when the editor's text changes; typically calls `db.update(...)` with a debounce.

<Note>
  `useCrdtField` must be called inside a `<Sp00kyProvider>` — it reads the active database from context.
</Note>

`getDoc()` returns a standard `LoroDoc`, so any Loro-compatible editor, data structure, or binding layer can plug in on top. Cursor presence is split into two hooks so you can bring your own editor:

- Call `pushCursorState(encoded)` whenever the local cursor moves — pass the bytes produced by Loro's `CursorEphemeralStore.encodeAll()`.
- Assign a callback to `onCursorUpdate` to receive the same bytes from every remote peer. Updates that arrive before the callback is set are replayed as soon as you assign it.

## How Syncing Works

CRDT state is persisted in a reserved `_00_crdt` table on the backend. When `useCrdtField` mounts, the client opens a `LIVE SELECT` on the relevant row and applies every incoming update to the local `LoroDoc`; local edits are encoded and written back the same way. Cursor presence rides on the same table under a `_cursor_<field>` key so it shares the same live connection. There is no extra configuration — any backend that supports live queries (self-hosted or Sp00ky Cloud) works out of the box.

Failed pushes are retried automatically (up to two times) before being logged, so transient network blips don't drop edits.

## Current Limitations

- **Text-only editor** — Only the `text` CRDT type has a built-in editor component (`CollaborativeEditor`). Other types require manual `LoroDoc` wiring.
- **SolidJS only** — First-class hooks ship in `@spooky-sync/client-solid`. The Flutter and Vanilla packages don't yet expose `useCrdtField`.
- **Live queries required** — Real-time merging depends on `LIVE SELECT`. Without live queries, edits only reconcile on the next full reload.
- **API stability** — Hook signatures and `CrdtField` methods may change while the feature is experimental.
