Sp00ky Cloud handles infrastructure provisioning, scaling, and management so you can deploy your project with a single command. No Docker Compose files, no server setup — just `spky cloud deploy`.

## Prerequisites

- The Sp00ky CLI installed (`npm i -g @sp00ky/cli`)
- A `sp00ky.yml` config file in your project root
- Docker installed locally (for building backend images)

## Quick Start

<CodeBlock
  code={`# 1. Log in to Sp00ky Cloud
spky cloud login

# 2. Create a cloud project
spky cloud create --slug my-app

# 3. Deploy
spky cloud deploy`}
  lang="bash"
/>

## Authentication

Log in via GitHub OAuth. The CLI opens your browser to complete the flow:

<CodeBlock
  code={`spky cloud login`}
  lang="bash"
/>

A 4-digit verification code is shown in your terminal. Confirm it in the browser to authenticate. Credentials are saved to `~/.sp00ky/credentials.json`.

To log out:

<CodeBlock
  code={`spky cloud logout`}
  lang="bash"
/>

<Note title="CI/CD">
For automated pipelines, create an API key with <code>spky cloud keys create</code> and set the <code>SPOOKY_API_KEY</code> environment variable. See [CI/CD](/docs/cloud/ci-cd) for details.
</Note>

## Creating a Project

<CodeBlock
  code={`# Interactive — prompts for slug if not provided
spky cloud create

# With options
spky cloud create --slug my-app --plan starter`}
  lang="bash"
/>

Available plans: `starter`, `pro`, `business`. The slug must be lowercase alphanumeric with hyphens, and is globally unique.

Your project slug should match the `slug` field in your `sp00ky.yml`:

<CodeBlock
  code={`# sp00ky.yml
slug: my-app
# ... rest of config`}
  lang="yaml"
/>

## Environment Variables

Every app in `sp00ky.yml` supports a unified `env` property for injecting environment variables into both local dev and cloud deployments. There are several ways to provide them.

### Dotenv File

Point to a dotenv file (one `KEY=VALUE` per line):

<CodeBlock
  code={`apps:
  api:
    type: backend
    env: ".env.local"
    # ...`}
  lang="yaml"
/>

### Inline Map

Pass variables directly in the config:

<CodeBlock
  code={`apps:
  api:
    type: backend
    env:
      DATABASE_URL: "postgres://localhost/mydb"
      LOG_LEVEL: debug
    # ...`}
  lang="yaml"
/>

### Per-Environment

Use different sources for local development (`spky dev`) vs cloud deployment (`spky cloud deploy`):

<CodeBlock
  code={`apps:
  agent:
    type: backend
    env:
      dev: ".env.local"       # loaded during spky dev
      cloud: "vault"           # loaded from encrypted vault on deploy
    # ...`}
  lang="yaml"
/>

### Encrypted Vault

Load all variables from the [Sp00ky Cloud vault](/docs/cloud/env-variables):

<CodeBlock
  code={`apps:
  api:
    type: backend
    env: "vault"
    # ...`}
  lang="yaml"
/>

To load only specific variables, use a whitelist:

<CodeBlock
  code={`apps:
  web:
    type: frontend
    env:
      vault: [VITE_API_URL, VITE_PUBLIC_KEY]
    # ...`}
  lang="yaml"
/>

<Note>
  Vault variables must be set up first with <code>spky cloud env init</code> and <code>spky cloud env set</code>. See [Environment Variables](/docs/cloud/env-variables) for the full workflow.
</Note>

### Merging Sources

Combine multiple sources in an array — later entries override earlier ones:

<CodeBlock
  code={`apps:
  api:
    type: backend
    env:
      - "vault"                        # base: all vault vars
      - ".env.overrides"               # override specific values from file
      - { LOG_LEVEL: debug }           # final inline override
    # ...`}
  lang="yaml"
/>

All forms can be combined with per-environment splits:

<CodeBlock
  code={`apps:
  agent:
    type: backend
    env:
      dev:
        - ".env.local"
        - { DEBUG: true }
      cloud:
        - "vault"
        - ".env.cloud.overrides"
    # ...`}
  lang="yaml"
/>

For the full env reference, see [Backend Functions — Environment Variables](/docs/backend-functions#environment-variables).

## Listing Projects

<CodeBlock
  code={`spky cloud list`}
  lang="bash"
/>

Shows all your cloud projects with their plan, status, and creation date.

## Destroying a Project

<Warning>
This permanently tears down all VMs and infrastructure for the project. Data will be lost unless you have a backup.
</Warning>

<CodeBlock
  code={`spky cloud destroy`}
  lang="bash"
/>
